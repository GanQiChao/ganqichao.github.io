sdp 用户故事

2017-11-06
project

sdp 索引导航

用户故事、需求详情、修改详情，细化业务用例内容，拆分每一项内容作为独立的工作登记记录，可以理解为任务分解。

需求详情 - Requirement Detail

项目 描述
标题 需求项
展示 分类+标题
分类 【需求】，标题的前缀，SDP系统提供选项
名称 英文或yyyyMMdd-01
描述 1）客户对该需求项的描述。
2）需求分析师对需求的理解和响应
引用 弱关系

需求项
需求文档具体的每项需求应当记录一个用户故事；另外需求分为【功能性需求】和【非功能性需求】，非功能型需求指响应时间等，建议在标题前缀添加【功能】与【非功能】标识

弱关系

  1. 若A与业务用例存在关系，则表示A的上文；格式为“（上文）【需求】业务用例-标题”
  2. 若A与用户故事存在关系，则表示A的下级；格式为“（下文）【需求】【功能】用户故事-标题”
  3. 若A与系统功能存在关系，则表示A的下级；格式为“（下文）系统功能-标题”

示例1

  • 标题：【需求】【功能】网格点分值计算
  • 名称：20171106-01
  • 描述：网格点是由网格大小将定级范围划分；分值计算是由因子树、计算方式按用途（商业、住宅、办公）分别计算各种用途下每个网格点的分值；因子树是由外部ArgGis工具制作图层shp文件导入。
  • 引用：
    1. （上文）【需求】业务用例-基准地价定级计算系统需求20171021
    2. （下文）【需求】【功能】因子因素体系及图层管理

用户故事【需求】还存在关键字说明，如上例的因子、图层。

示例2

  • 标题：【需求】关键字说明
  • 名称：Keyword
  • 描述：
    1. 因子…
    2. 因素…
    3. 图层…
  • 引用：无

修改详情 - Change Detail

项目 描述
标题 修改项，参照“用户故事【需求】，标题”
应当严格按照引用业务用例划分为【修改】
名称 英文
描述 资料文件
引用 参照“用户故事【修改】”

用户故事 - User Story

项目 描述
标题 执行者/业务工人/业务实体+场景概述
名称 英文
故事 叙述现状和改进，现状将严格依据客户当前实际情况。
改进将采用步骤方式说明执行者、业务工人、
业务实体和待研发系统之间是如何协调以完成工作。
引用 参照《用户故事【需求】》

标题

  1. 业务工人， 与系统直接打交道的组织内的人
  2. 业务实体， 非人，系统、时间等
  3. 场景概述，概述故事

示例

  • 标题：【业务】数据人员计算网格点分值成果
  • 名称：Data manager get grid point score result
  • 描述：
    1. 数据人员建立因子树
    2. 数据人员导入图层
    3. 数据人员建立网格
    4. 数据人员提交计算方式
    5. 系统计算并记录网格点分值
    6. 数据人员下载网格点分值成果shp
  • 引用：
    1. （上文）【业务】评估部门获取网格点分值成果
    2. （下文）【业务】数据人员建立因子树
    3. （下文）【业务】数据人员导入图层
    4. （下文）【业务】数据人员建立网格
    5. （下文）【业务】数据人员提交计算方式
    6. （下文）【业务】系统计算并记录网格点分值

（下文）【业务】数据人员建立因子树

示例：

  • 标题：【业务】数据人员建立因子树
  • 名称：Data manager build factor tree
  • 描述：（假设不存在现状，直接叙述故事步骤）
    1. 数据人员添加因素和因子
    2. 数据人员修改因素和因子
    3. 数据人员删除错误因素和因子
    4. 数据人员验证因子数据
    5. 略
  • 引用：
    1. （上文）【业务】数据人员计算网格点分值成果
